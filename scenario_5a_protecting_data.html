Initial thoughts
Linux File Permissions — Reference
Everything you need to configure access controls correctly. Study this before working on the lab.

Permission Types

rRead — view contents of a file or list a directory
wWrite — modify a file or create/delete files in a directory
xExecute — run a file as a program, or enter (cd into) a directory
-No permission for that action

Who the Permissions Apply To

uUser — the owner of the file
gGroup — users in the file's assigned group
oOthers — everyone else on the system
aAll — user + group + others combined

Octal (Numeric) Values

4Read (r)
2Write (w)
1Execute (x)
7rwx (4+2+1 = full access)
5r-x (4+0+1 = read & execute)
4r-- (read only)
0--- (no access)

Reading a Permission String

drwxr-x---
dDirectory (d) or file (-)
rwxOwner: read, write, execute = 7
r-xGroup: read, execute = 5
---Others: no access = 0
The chmod command
sudo chmod -R 750 /RandD/Formula
sudoRun as superuser (required for system directories)
chmodChange mode — the command to set permissions
-RRecursive — apply to all files inside the directory too
750Octal code: owner=7 (rwx), group=5 (r-x), others=0 (---)
/pathThe directory or file to change
Common octal codes for this scenario
777
rwxrwxrwx — full access for everyone
774
rwxrwxr-- — owner & group full, others read-only
770
rwxrwx--- — owner & group full, others no access
750
rwxr-x--- — owner full, group r-x, others none
⚠ In Linux, permissions are set for user (owner), group, and others — in that order. For this scenario, the head researcher is the owner (user) of the Formula directory, and the R&D team is the group.

Ubuntu Linux — Hands-On Setup
Now run the real commands in your Ubuntu terminal. Work through each step in order and record your output.
⚠ Only run these commands in your classroom lab environment or a VM your teacher has set up. Do not run on a personal or shared production machine without authorization.
Step 1
Create the Groups
›

Before setting permissions, the randd group must exist on the system. Linux permissions work at the group level — every user assigned to randd automatically inherits the group permissions you set.

Commands to run
$ sudo groupadd randd
# Creates the R&D team group
$ getent group randd
# Verify the group was created — should show: randd:x:NNNN:
Step 2
Create the Users
›

Create the head researcher (who will own the directories) and a sample R&D team member. The -m flag creates a home directory; -g sets the primary group; -G adds secondary groups.

Commands to run
$ sudo useradd -m -g randd headresearcher
# Creates head researcher with randd as primary group
$ sudo useradd -m -g randd researcher1
# Creates a sample R&D team member
$ sudo useradd -m itstaff1
# Creates an IT staff user NOT in the randd group
$ id headresearcher
# Verify: should show uid, gid=randd, groups=randd
Step 3
Create the Directory Structure
›

Create the full directory tree from the scenario. The -p flag creates all parent directories in one command. Then create some placeholder files inside each directory to test permissions against.

Commands to run
$ sudo mkdir -p /RandD/Research /RandD/Formula
# Creates all three directories at once
$ sudo touch /RandD/Research/strype_trials.csv
$ sudo touch /RandD/Research/clinical_data.xlsx
$ sudo touch /RandD/Formula/compound_formula.txt
$ sudo touch /RandD/Formula/synthesis_process.pdf
# Creates placeholder research files
$ ls -la /RandD/
# List directories — note default permissions before we change them
Step 4
Set Ownership
›

Before permissions mean anything, ownership must be correct. Use chown to set the owner (user) and group for each directory. The format is chown owner:group path. The head researcher should own everything; the group should be randd.

Commands to run
$ sudo chown -R headresearcher:randd /RandD
# Sets owner=headresearcher, group=randd on all files recursively
$ ls -la /RandD/
# Verify ownership changed
$ ls -la /RandD/Formula/
# Check files inside Formula too
Step 5
Apply the chmod Permissions
›

Now apply the three chmod commands you worked out in the Permissions Lab above. Run them in order — parent directory first, then subdirectories.

Commands to run — use your octal codes from the lab
$ sudo chmod -R 774 /RandD
# Req 1: R&D full (7), others read-only (4)
$ sudo chmod -R 770 /RandD/Research
# Req 2: R&D full (7), others none (0)
$ sudo chmod -R 750 /RandD/Formula
# Req 3: Owner rwx (7), R&D r-x (5), others none (0)
$ ls -la /RandD/ && ls -la /RandD/Research/ && ls -la /RandD/Formula/
# Verify all three permission sets
Step 6
Test the Permissions
›

Testing is a critical part of security configuration — never assume permissions work, always verify. Switch between users and try to access each directory. A correctly configured system will allow and deny access exactly as specified.

Test as researcher1 (R&D team member)
$ sudo -u researcher1 ls /RandD/Research/
# Expected: lists files — R&D has full access
$ sudo -u researcher1 cat /RandD/Formula/compound_formula.txt
# Expected: can read — R&D has r-x on Formula
$ sudo -u researcher1 touch /RandD/Formula/test.txt
# Expected: Permission denied — R&D cannot write to Formula
Test as itstaff1 (not in randd group)
$ sudo -u itstaff1 ls /RandD/
# Expected: lists top-level contents — others have read on /RandD
$ sudo -u itstaff1 ls /RandD/Research/
# Expected: Permission denied — others have no access to Research
$ sudo -u itstaff1 ls /RandD/Formula/
# Expected: Permission denied — others have no access to Formula
Step 7
Verify with stat & getfacl
›

Security engineers document their configurations. The stat command shows detailed file metadata including exact octal permissions. getfacl shows the full Access Control List in human-readable form — this is what you'd include in a security audit report.

Commands to run
$ stat /RandD /RandD/Research /RandD/Formula
# Shows octal permission codes — look for "Access: (0774/drwxrwxr--)"
$ getfacl /RandD /RandD/Research /RandD/Formula
# Shows full ACL — owner, group, other in readable format

Discussion
Connect this scenario to broader questions about data security, access control, and professional responsibility.
💬 Think beyond the commands. Access control is a policy decision as much as a technical one — who decides who gets access, and what happens when that decision is wrong?
Q1 — Least Privilege
The principle of "least privilege" means giving users only the access they need to do their job — nothing more. How does this scenario apply that principle? Can you think of a real-world example where too much access caused a breach?
Q2 — Insider Threats
Even with correct permissions, a member of the R&D team could copy the Formula files to a USB drive. What additional controls — technical or managerial — could Vitahealth implement to prevent insider data theft?
Q3 — Air Gaps vs. Connectivity
Air-gapping prevents remote attacks but creates operational challenges — patches must be applied manually, data must be transferred physically. What are the tradeoffs? When is an air gap worth it?
